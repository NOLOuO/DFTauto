# 文献评分机制说明

本文档对应当前程序里的实际实现，代码来源是 [score_refs.py](F:\DFTauto\literature-review\scripts\score_refs.py)。

目的不是给出“理想方案”，而是准确说明：

- 现在程序到底怎么打分
- 每一项分数怎么来的
- 当前机制的优点和偏差
- 哪些地方最值得改

---

## 1. 当前评分目标

当前评分机制服务于一个很具体的目标：

从“已经读过的论文”的参考文献中，筛出下一轮最值得优先下载和阅读的候选论文。

所以它不是学术搜索引擎式的“全网排序”，而是一个面向滚雪球扩展的启发式优先级模型。

---

## 2. 当前输入与输出

### 输入

输入是已经结构化完成的论文 JSON，通常来自：

- `output/parsed/seed_info/*.json`
- `output/parsed/round_1_info/*.json`
- `output/parsed/round_2_info/*.json`

这些 JSON 中至少包含：

- 当前论文标题
- DOI
- 参考文献列表
- 参考文献标题
- 参考文献 DOI
- 年份
- 期刊
- 第一作者

### 输出

程序输出两个文件：

1. 待读队列 CSV
   - 如 `output/records/queue/round_1_reading_queue.csv`

2. 下载日志初始化 CSV
   - 如 `output/records/queue/round_1_download_log.csv`

其中待读队列 CSV 只保留前 `top_n` 篇高优先级候选。

---

## 3. 当前去重逻辑

程序会先聚合所有已读论文中的参考文献，然后做去重。

### 去重键

优先使用 DOI：

- 如果参考文献有 DOI，则 key 为 `doi:{doi}`

否则退化为标题：

- `title:{规范化标题}`

### 不再重复纳入候选的条件

如果候选论文满足以下任一条件，就会被跳过：

- DOI 已经出现在已读论文集合中
- 规范化标题已经出现在已读论文集合中

这意味着：

- 已经读过的论文不会再次进入候选
- 但“上一轮候选过、却没下载成功的论文”，仍可能在后续轮次再次出现

---

## 4. 总分公式

当前总分公式是：

```text
总分 = 共被引分 + 相关性分 + 影响力分 + 新近性分 + 可获取性分
```

最终分数保留两位小数。

---

## 5. 各分项细节

### 5.1 共被引分 `score_cocitation`

定义：

- 有多少篇“已读论文”同时引用了这篇候选文献

公式：

```text
score_cocitation = min(cited_by_count * 2.5, 5.0)
```

也就是：

- 被 1 篇已读论文引用：2.5 分
- 被 2 篇已读论文引用：5.0 分
- 被 3 篇及以上引用：仍然封顶 5.0 分

解释：

- 这是当前最强的信号之一
- 它认为“多篇已读论文共同引用的文献”更值得优先扩展

优点：

- 非常适合滚雪球扩展
- 对核心基础论文比较敏感

问题：

- 2 次共被引就封顶，区分度偏低
- 容易偏向“老而经典”的公共参考文献

---

### 5.2 相关性分 `score_relevance`

定义：

- 看标题和期刊名里是否包含预设关键词

当前关键词及权重：

```text
agent: 1.6
agentic: 1.6
multi-agent: 1.8
autonomous: 1.4
llm: 1.5
large language model: 1.5
workflow: 1.2
automation: 1.3
dft: 1.5
density functional theory: 1.5
vasp: 1.1
quantum espresso: 1.1
hpc: 0.9
convergence: 0.9
materials: 0.5
```

公式：

```text
score_relevance = min(命中关键词权重之和, 4.0)
```

解释：

- 只要标题或期刊里含这些词，就累加对应权重
- 最多加到 4 分

优点：

- 简单直接
- 适合快速偏向“Agent + DFT + 自动化”主题

问题：

- 依赖字面关键词，语义覆盖较弱
- 容易漏掉“相关但不用这些词表达”的论文
- `journal` 也参与匹配，可能引入噪声

---

### 5.3 影响力分 `score_impact`

定义：

- 用 DOI、期刊名、共被引次数做一个非常轻量的影响力近似

公式逻辑：

基础分：

```text
0.5
```

如果有 DOI：

```text
+0.8
```

如果期刊属于高影响力列表：

```text
+1.0
```

如果共被引数 >= 3：

```text
+0.7
```

如果共被引数 == 2：

```text
+0.4
```

总分封顶：

```text
score_impact = min(上述和, 2.5)
```

当前高影响力期刊关键词列表：

```text
nature
science
pnas
npj
physical review
journal of the american chemical society
digital discovery
machine learning: science and technology
```

优点：

- 成本低
- 对明显高影响力来源有基本偏置

问题：

- 很粗糙，不等于真实学术影响力
- 依赖期刊字符串匹配，容易误伤或漏掉
- 会议、预印本、交叉学科论文的处理不够稳

---

### 5.4 新近性分 `score_recency`

定义：

- 按年份给分

规则：

```text
>= 2025 -> 2.0
>= 2023 -> 1.6
>= 2020 -> 1.2
>= 2015 -> 0.8
更早 -> 0.5
年份缺失 -> 0.4
```

优点：

- 明确鼓励新论文
- 对 Agent / LLM 这类快速演化主题有意义

问题：

- 对经典基础方法论文不够友好
- 年份带来的偏置比较强

---

### 5.5 可获取性分 `score_accessibility`

定义：

- 估计这篇论文是否更容易通过开放渠道下载到

规则：

```text
标题/期刊中含 arxiv 或 chemrxiv -> 1.8
有 DOI -> 1.2
只有期刊信息 -> 0.8
都缺 -> 0.4
```

优点：

- 能提升自动化流程的实际跑通率
- 更偏向可下载的论文，而不是只偏向“理论上重要”

问题：

- “有 DOI”不等于“容易下载”
- 这部分其实更像执行便利性，而不是学术价值

---

## 6. 优先级标签

总分算完后，会打上三个优先级标签：

```text
score >= 10    -> must_read
score >= 8     -> recommended
否则           -> backlog
```

排序规则是：

1. `must_read` 优先于 `recommended`
2. `recommended` 优先于 `backlog`
3. 同优先级内按总分降序
4. 再按标题字母序

然后只保留前 `top_n` 篇。

---

## 7. 当前机制的整体倾向

如果把它概括成一句话：

当前评分机制偏向于选出“被多篇已读论文共同引用、标题看起来和 Agent/DFT/自动化相关、而且较新且较容易下载”的论文。

更具体地说，它会偏向：

- 多篇种子论文共同引用的核心文献
- 标题里有 `agent`、`llm`、`workflow`、`automation`、`dft` 的论文
- 近几年论文
- 带 DOI、预印本、容易 OA 下载的论文

它会相对吃亏的文献类型包括：

- 标题不显式写关键词，但内容很相关的论文
- 经典老论文
- 非常重要但下载渠道较差的论文
- 跨学科表达方式不同的论文

## 8. 我认为最值得优先改的地方

下面这些是最可能影响结果质量的部分。

### 方案 A：降低“关键词匹配”的权重

建议：

- 适当降低 `score_relevance` 上限
- 或把部分关键词权重调低

原因：

- 当前关键词分有时会把“标题写得像目标方向”的论文抬得过高

---

### 方案 B：提高共被引分的区分度

建议：

- 不要在 `2` 次共被引时就封顶
- 可以改成渐进式增长，例如上限 `6` 或 `7`

原因：

- 现在 `2` 次和 `6` 次共被引没有差别，不利于识别真正的核心枢纽文献

---

### 方案 C：把“可获取性”从总分里剥离

建议：

- 学术排序和下载便利性分开
- 总分先按学术相关性排序，再把下载便利性作为并列参考字段

原因：

- “容易下载”不应过度影响“是否值得读”

---

### 方案 D：增加“主题匹配层级”

建议：

把文献分成几类再打分，例如：

- Agent/LLM 驱动科学自动化
- DFT / electronic structure automation
- workflow / orchestration / fault recovery
- HPC / scheduler / infrastructure

原因：

- 你的项目不是一般材料文献综述，而是“Agent for DFT automation”
- 单靠关键词加权还不够细

---

### 方案 E：给老经典论文保留入口

建议：

- 对高共被引但年份较老的论文设保底机制
- 或把“经典方法论文”单独打一类标签

原因：

- 否则经典基础设施论文会被“新近性”压低

---

## 9. 我建议的下一版方向

如果你让我改，我建议优先改成下面这个思路：

### 新版原则

总分分成两部分：

```text
学术价值分 + 执行便利分
```

其中：

- 学术价值分
  - 共被引
  - 主题相关性
  - 影响力
  - 新近性

- 执行便利分
  - 可获取性

最后输出两列：

- `score_academic`
- `score_operational`

再综合得到：

- `score_total`

这样排序会更清楚，也更容易解释给别人。

---

## 10. 你可以重点决定的修改项

你看这份机制时，建议优先决定下面几件事：

1. `共被引分` 要不要继续当第一大信号
2. `相关性关键词` 要不要调整词表和权重
3. `新近性` 是否对老论文过于苛刻
4. `可获取性` 是否应该继续并入总分
5. `must_read / recommended / backlog` 的阈值是否要改
6. 是否需要给不同主题设不同评分通道

---

## 11. 我对当前机制的简短判断

当前机制适合作为第一版自动扩展器，优点是：

- 简单
- 可解释
- 成本低
- 跑得稳

但如果目标是“给别人用”或者“发布程序”，它还比较像：

- 工程可用的启发式规则

而不是：

- 更可靠的研究级排序系统

所以它现在能用，但确实值得再打磨一轮。
